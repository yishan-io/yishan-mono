# Contributing

Thanks for contributing to YISHAN. This guide is the fastest path to getting a change merged.

## Contributor Expectations

- Keep PRs focused on one problem.
- Align scope first for non-trivial work (open an issue before coding).
- Follow repository engineering rules in `AGENTS.md`.
- Follow our [Code of Conduct](CODE_OF_CONDUCT.md).

## Quick Start

```bash
bun install
bun run prepare
```

Key workspace commands:

```bash
bun run typecheck
bun run lint
bun run test:unit
bun run test:e2e
bun run build
```

Git hooks (`simple-git-hooks`) run automatically:

- `pre-commit`: `bun run format && bun run typecheck`
- `pre-push`: `bun run lint && bun run test:unit && bun run build`

## Contribution Workflow

1. Create or link an issue.
2. Define scope clearly (in scope and out of scope).
3. Implement the smallest complete change.
4. Add or update tests for behavior changes.
5. Open a PR using `.github/pull_request_template.md`.

Use `.github/ISSUE_TEMPLATE/work-item.md` when opening a new issue.

## Recommended: Skill-Assisted Issue and PR Flow

To standardize issue and PR quality, we recommend using repository skills when available:

- `create-github-issue`: creates issues aligned with `.github/ISSUE_TEMPLATE/work-item.md`.
- `pr-skill`: creates PRs aligned with `.github/pull_request_template.md` and required linkage.

Manual issue/PR creation on GitHub remains fully supported.

## Quality Bar (Required)

A contribution is considered ready only when all of these are true:

- Behavior changes include relevant tests.
- Lint and typecheck pass.
- Doc comments are updated for exported and non-trivial functions you changed.
- Scope matches the linked issue/plan.

Required local validation before pushing:

```bash
bun run typecheck
bun run lint
bun run test:unit
```

Run `bun run test:e2e` when your change affects desktop runtime flows or UI behavior.

## Pull Request Requirements

- Fill all sections of `.github/pull_request_template.md`.
- Link issue(s) using `closes #...` / `fixes #...` when applicable.
- Include testing notes with commands and outcomes.
- Choose exactly one release-notes label:
  - `release:feature`
  - `release:fix`
  - `release:improvement`
  - `release:docs`
  - `skip-release-notes` for internal-only changes

## Release Notes and Versioning

- `CHANGELOG.md` is not maintained manually.
- Release notes are generated by GitHub Releases (`--generate-notes`), configured by `.github/release.yml`.
- Maintainers trigger `.github/workflows/release.yml` for release prep/tagging/publishing.

## Agent Automation

- Shared automation scripts live in `scripts/automation/`.
- Canonical skill docs live in `.agents/skills`.
- Regenerate wrappers with `bun run skills:sync`.
- Do not duplicate logic in per-agent wrappers when shared scripts already exist.
- Details: `docs/agent-automation.md`.
